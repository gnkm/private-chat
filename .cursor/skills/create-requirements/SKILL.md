# Create Requirement Skill

ISO/IEC/IEEE 29148:2018（*Systems and software engineering — Life cycle processes — Requirements engineering*）に整合する **ソフトウェア要求仕様書（SRS）** を書くための手順。標準は要求プロセス・情報項目の内容・書式の指針を定める。本スキルは特に **SRS 情報項目（同標準 Clause 9.6 付近）** と **個別要求の品質** に焦点を当てる。

## When to Use

- 新規プロダクト／機能の **SRS** をゼロから起こす、または既存 SRS を改訂する
- 要求を **ステークホルダ要求（StRS）・システム要求（SyRS）** から **ソフトウェア要求（SRS）** へ落とし込み、境界とトレーサビリティを明確にしたい
- 要求が **曖昧・重複・検証不能** にならないようレビュー観点を揃えたい
- ユーザーが「29148 に沿った要求仕様」と明示したとき

## Instructions

### 1. スコープと前提の確認

- **対象**: ソフトウェア境界（何がソフトウェアの責務で、何がハード／運用／他システムか）。
- **上流との関係**: 29148 が示す要求情報項目の階層（例: **BRS → StRS → SyRS → SRS**）のうち、既に存在する文書があるなら **SRS はソフトウェアに割り当てられた要求** に集中し、上流 ID への参照またはトレースを残す。
- **未確定事項**: 仕様に穴がある場合はユーザーに質問する（Cursor の質問 UI 等）。推測で埋めない。TBR（To be resolved）を置く場合は **所有者・解消条件** を添える。

### 2. 個別要求の記述ルール（well-formed 要求）

各要求は **一意の識別子**（例: `SRS-FUNC-001`）を付与し、次の観点で書く（29148 が求める「よい要求」の特性に整合するよう心がける）。

| 観点 | エージェントが守ること |
|------|------------------------|
| 単一・明確 | 1 文で 1 要求。曖昧語（「適切」「高速」「十分」）は **定量化または判定基準** に置き換える |
| 検証可能 | **受入・テスト・計測**で真偽を判定できる表現（Given/When/Then、計測名と閾値、状態遷移など） |
| 実現可能 | 制約（法令、セキュリティ、互換性）と矛盾しない |
| トレーサブル | 上流要求 ID、設計・試験へのリンク欄を確保（表またはセクション） |
| 完全 | 入力・前提・例外・エラー動作まで必要な範囲で記述 |

### 3. SRS の文書構成（標準の SRS 情報項目・付属の例示アウトラインに整合）

出力はリポジトリの慣例に合わせる（例: `docs/requirements.md`）。少なくとも次の **論理ブロック** を満たすこと。見出し名はプロジェクトで調整してよいが、**内容相当性** を失わない。

1. **文書管理情報**  
   版、日付、作成者／レビュー、変更履歴、適用する標準（本書は 29148:2018 準拠意図）。
2. **はじめに**  
   **目的**（SRS の読者と用途）、**範囲**（対象ソフトウェアと境界）、**用語・略語**、**参照**（関連 StRS/SyRS、ConOps、外部規格）。
3. **全体像**  
   運用コンテキスト、主要ステークホルダ、想定利用形態、**前提・依存**、**制約**（技術・プロジェクト・規制）。
4. **外部インタフェース要求**  
   ユーザー UI、API、ファイル、他システム、ハード／OS との契約（データ項目、エラー、性能のインタフェース面）。
5. **機能要求**  
   ユースケース／ユーザーストーリー／機能一覧に整理し、各条を **識別子付きの単独要求** で記述。
6. **非機能要求（ソフトウェアシステム属性）**  
   可用性、性能、セキュリティ、保守性、移植性など。**測定可能なしきい値** を付ける。
7. **設計・実装上の制約**  
   言語／フレームワーク／認証方式など「実装の自由度を狭める」事項を要求として明示（根拠があれば簡潔に）。
8. **要求の配分・除外**  
   システム要求のうち本ソフトウェアに属さないもの、将来フェーズに回すものを明示（標準が重視する **要求の配分（apportioning）** の考え方に沿う）。
9. **検証**  
   各要求または要求グループに対する **検証方法**（試験、分析、実演、検査）の方針を対応表で示す。

### 4. トレーサビリティとレビュー

- **トレーサビリティ行列（RTM）** または同等の表で、上流 ID ↔ SRS ID ↔ 試験ケース（または検証観点）を追えるようにする。
- 初版作成後、**曖昧さスキャン**（曖昧語、未数値化の性能、未定義のエラー処理）と **重複・矛盾チェック** を行い、必要ならユーザーに確認質問を返す。

### 5. ユーザーとの対話

- ビジネス目的、優先度、非対象、規制・SLA が不明なときは **先に質問** してから本文を確定する。
- ユーザーが「この章だけ」「API だけ」など部分スコープを指定した場合は、その範囲に限定しつつ **文書全体との整合**（参照・用語の一貫性）を壊さない。
