# TDD ワークフロー（Private Chat）

## When to Use

- コードを追加・変更する **前** にテストの置き場所と失敗条件を決められるとき
- ユーザーが **TDD**、**テスト先行**、**Red-Green** と言及したとき
- `AGENTS.md` の開発ルール（TDD 徹底）に従う必要があるとき

**使わない**: 純ドキュメントのみの変更、設定ファイルのコメントのみ、生成物の機械的コピーなど、テスト対象がない変更。

## 正本との関係

- **`AGENTS.md`**: 「コードを書く前に必ずテスト」「Red → Green → Refactor」を最優先する。
- **`docs/tech-stack.md`**: 単体・コンポーネントは **Vitest**、E2E は **Playwright**（`e2e/` 集約が想定）。
- **`docs/architecture.md`**: `packages/shared` が Zod／型の共有の正本である場合、**契約のテストは共有層または両者で重複しないよう** 一箇所に寄せる。

## コアサイクル（必須）

各変更について、次の順序を **飛ばさない**。

1. **Red**: **まだ存在しない振る舞い**、または **バグの再現** を表すテストを書き、**意図的に失敗** させる（コンパイルエラーも Red でよい）。
2. **Green**: そのテストを通す **最小限の実装** のみ行う。余計な機能・リファクタを混ぜない。
3. **Refactor**: テストが緑のまま、重複除去・命名・分割を行う。振る舞いを変えるなら **新しい Red** からやり直す。

## テストの置き場（目安）

| 対象 | 主なツール | 置き場の目安 |
|------|------------|----------------|
| 共有スキーマ・純関数・ドメイン | Vitest | `packages/shared` または該当パッケージ内 |
| React コンポーネント・フック | Vitest + Testing Library 等 | `apps/web` 近傍 |
| Express / WebSocket ハンドラ | Vitest | `apps/server` 近傍 |
| ブラウザ越しのクリティカルパス | Playwright | `e2e/` |

E2E は遅く・不安定になりやすい。**同じ振る舞いは可能なら Vitest で先に固定**し、E2E はユーザー旅程の要所に絞る。

## 実装順序の原則

- **契約（JSON／Zod／型）が先**: サーバとクライアントで同じペイロードを扱う場合、`packages/shared` のスキーマと、そのスキーマに対するテストを **先に** Red→Green し、あとから HTTP/WebSocket に接続する。
- **バグ修正**: まず **失敗する回帰テスト**（最小再現）を Red にする。再現できない場合は、ログや仮説をユーザーに確認してから着手してよい。
- **大きい機能**: 縦に薄いスライス（例: 「検証失敗時にクライアントへエラーが返る」まで）で Red→Green→Refactor を **複数回** 回す。一気に全部のテストを書いてから実装しない（千行 Red は避ける）。

## Red の品質

- 失敗理由が **意図したもの** であることを確認する（例: `AssertionError` でなく `TypeError` ばかりならテストが不十分）。
- テスト名・コメントで **どの要求や不具合に対応するか** を一言で残すとよい（SRS ID や Issue が分かれば記載）。

## Refactor で守ること

- **Green のまま** 小さく刻む。テストを緩めて緑にするのは **最後の手段** であり、その場合は理由を明記する。
- プロジェクトの **Lint / 型**（Biome、`tsc`）を壊さない。

## アンチパターン（避ける）

- 「実装だけ先に書き、あとでテストを足す」
- Red を確認せずに実装を大量投入する
- E2E だけで単体可能なロジックをすべてカバーする
- `packages/shared` を経由せず、フロントとサーバで別々の型だけ書く（ズレの温床）

## コミットの粒度（推奨）

- **Red** のコミットと **Green** のコミットを分けられるとレビューしやすい。無理なら「テスト追加 + 実装」を一コミットにまとめ、PR 説明で Red→Green が追えるようにする。
