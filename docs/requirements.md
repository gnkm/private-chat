## 2. はじめに

### 2.1 目的

本書は、閉じたネットワーク内で利用する **Slack ライクなユーザーインタフェースを持つ簡易チャットアプリケーション** に対して、開発・試験・受入判定の根拠となるソフトウェア要求を定義する。

**想定読者**: 開発者、試験担当者、プロダクトオーナー。

### 2.2 範囲

**対象ソフトウェア**: チャット画面の提示、投稿の送受信、およびそれを支えるクライアント／サーバのソフトウェア。

**ソフトウェア境界**:

- **本書の対象**: ブラウザ上の UI、メッセージ投稿 API／リアルタイム配信のクライアント／サーバ実装、およびクライアント側でのメッセージ一覧の揮発的な保持。
- **対象外**: ユーザー登録・認証基盤、永続データベース、運用監視ツール本体、ネットワーク機器の設定。

### 2.3 用語・略語

| 用語 | 定義 |
|------|------|
| 投稿 | 利用者が入力し送信した 1 件のテキストメッセージ |
| 表示名 | 画面上に投稿者として表示される文字列（アカウントとは紐付けない） |
| 閉じたネットワーク | インターネット全体に公開せず、特定の LAN／VPN 等のみで到達可能な環境 |

### 2.4 参照

- ISO/IEC/IEEE 29148:2018 — *Systems and software engineering — Life cycle processes — Requirements engineering*
- 上流の別紙 StRS／SyRS は現時点では未作成のため、本 SRS が前提要求の一次情報源とする。

## 3. 全体像

### 3.1 運用コンテキスト

利用者は同一の閉じたネットワーク上から Web ブラウザでアプリケーションにアクセスし、単一の会話スペース（単一チャンネル相当）に投稿する。サーバプロセスは当該ネットワーク内のホストで稼働する。

### 3.2 ステークホルダ

| ステークホルダ | 利害・関心事 |
|----------------|--------------|
| エンドユーザー | メッセージをすばやく読み書きできること |
| 開発／運用 | 構成が単純で、DB や認証基盤を不要にできること |

### 3.3 前提・依存

- 利用者は対象ネットワークに既に接続済みである（VPN 構成そのものは本ソフトウェアの責務外）。
- クライアントは **モダンな Web ブラウザ**（ECMAScript と WebSocket またはサーバ送信イベント等、選定したリアルタイム手段をサポートすること）を利用する。

### 3.4 制約（プロジェクト／技術）

- **データベースを使用しない**（永続ストアとしての RDB／NoSQL は採用しない）。
- **ユーザー登録およびログイン機構を実装しない**。
- **機能スコープは投稿に限定する**（編集・削除・ファイル添付・スレッド等は本リリースの対象外とする。詳細は §8）。

## 4. 外部インタフェース要求

### 4.1 ユーザーインタフェース（Slack ライク）

| ID | 要求 |
|----|------|
| SRS-UI-001 | メイン画面は、中央に投稿一覧と入力欄を備えた **Slack に類似したレイアウト** とする。 |
| SRS-UI-002 | 投稿一覧は **時系列（古いものが上）** で表示し、各投稿に **表示名** と **本文** と **送信時刻** を表示する。 |
| SRS-UI-003 | 利用者は **テキスト入力欄** と **送信操作**（ボタンまたは Enter 送信のいずれか、実装で固定）により投稿できる。 |
| SRS-UI-004 | 表示名は **ブラウザに保存される任意文字列**（ローカルストレージ等）により設定でき、サーバ側アカウントとは紐付けない。 |

### 4.2 ソフトウェア間インタフェース（API／プロトコル）

| ID | 要求 |
|----|------|
| SRS-IF-001 | サーバは **投稿の受付** と **他クライアントへの配信** を提供するインタフェースを公開する（具体的なプロトコルは設計書で定義）。 |
| SRS-IF-002 | 認証ヘッダやセッション Cookie に依存した **本人確認を要求しない**。 |

### 4.3 エラー表示（インタフェース面）

| ID | 要求 |
|----|------|
| SRS-IF-003 | 送信に失敗した場合、利用者が原因区分を理解できる **簡潔なエラー表示**（画面上のメッセージ）を行う。 |

## 5. 機能要求

### 5.1 投稿の送信

| ID | 要求 |
|----|------|
| SRS-FUNC-001 | 利用者が入力した本文が空でないとき、送信操作により **サーバへ投稿が送信** される。 |
| SRS-FUNC-002 | サーバは受信した投稿を **同一セッションの全接続クライアントに配信** する（単一会話スペース）。 |
| SRS-FUNC-003 | 各投稿には **サーバが付与または合意した一意の識別子** および **送信日時（タイムスタンプ）** が付く。 |

### 5.2 投稿の表示

| SRS-FUNC-004 | クライアントはサーバから受信した投稿を **一覧に追加表示** し、SRS-UI-002 に従って表示する。 |

### 5.3 メッセージ履歴の扱い（DB 不使用）

| SRS-FUNC-005 | サーバは投稿本文を **一覧として蓄積保持しない**。各投稿の **一意な識別子および送信日時（タイムスタンプ）はサーバが受付時に付与** し、接続中のクライアントへブロードキャストする。**一覧としての投稿の保持場所は各クライアントの表示状態** とする（永続 DB は用いない）。 |
| SRS-FUNC-006 | **サーバプロセス再起動後**、過去投稿は **サーバから復元されない**（仕様として許容する）。再接続したクライアントは、その時点以降にブロードキャストされる投稿のみを受け取れる。 |

### 5.4 投稿本文の形式制約（TBR-002 解消）

| ID | 要求 |
|----|------|
| SRS-FUNC-007 | 投稿本文の長さは **最大 12,000 文字** とする。文字の数え方は **Unicode コードポイント** を単位とし、実装（Zod・クライアント入力）で同一の定義に揃える。 |
| SRS-FUNC-008 | 投稿本文に **U+0000（NUL）** を含めてはならない。改行（U+000A LF、U+000D CR）およびタブ（U+0009）は **許可** する。表示における XSS 対策のため HTML 記号等を禁止する文字列フィルタは **行わない**（ブラウザ側の通常のテキスト表示に依拠する）。 |

## 6. 非機能要求（ソフトウェアシステム属性）

| ID | 区分 | 要求 |
|----|------|------|
| SRS-NF-001 | 性能（応答） | 同一 LAN 内、通常負荷（同時利用者 50 名未満を想定）において、投稿送信から当該クライアントの一覧反映まで **2 秒以内** を目標とする（計測はブラウザの開発者ツールまたはログタイムスタンプで検証可能とする）。 |
| SRS-NF-002 | 可用性 | 単一サーバプロセス構成とし、プロセス停止時はサービス利用不可となることを許容する（高可用構成は対象外）。 |
| SRS-NF-003 | セキュリティ（前提） | 本ソフトウェア単体では **認証・アクセス制御を提供しない**。閉じたネットワークでの利用を前提とし、ネットワーク境界での制御は運用側の責務とする。 |
| SRS-NF-004 | 保守性 | サーバおよびクライアントのビルド・起動手順は README に記載し、**新規開発者がリポジトリのみで起動確認**できること。 |

## 7. 設計・実装上の制約

| ID | 要求 |
|----|------|
| SRS-CSTR-001 | **永続データベース**（ファイルを除く DBMS）を **使用しない**。 |
| SRS-CSTR-002 | **ユーザー登録・ログイン** の機能を **実装しない**。 |
| SRS-CSTR-003 | 機能は **投稿の送受信と表示** に限定する（§8 の除外機能を実装しない）。 |

---

## 8. 要求の配分・除外（本ソフトウェアに含めないもの）

以下は **本リリースの SRS 対象外** とする。

- ユーザー認証、ロール／権限管理
- RDB／NoSQL 等によるメッセージの永続化
- メッセージの編集・削除、リアクション、メンション、チャンネル複数化、ダイレクトメッセージ
- ファイル・画像の添付
- 検索、通知（デスクトップ／メール）、メッセージのインポート／エクスポート
- 監査ログの長期保管（必要になった場合は別要求として追加する）

将来これらを追加する場合は、上流要求の更新および本 SRS の改訂版を作成する。

## 9. 検証

### 9.1 検証方法の対応（抜粋）

| 要求 ID | 検証方法 | 検証観点（例） |
|---------|----------|----------------|
| SRS-UI-001〜004 | 実演・検査 | レイアウト、表示項目、表示名設定の有無 |
| SRS-FUNC-001〜004 | 試験（手動または E2E） | 送信、配信、一覧の更新 |
| SRS-FUNC-007〜008 | 試験・検査 | 最大長・NUL 拒否・改行許可の検証 |
| SRS-FUNC-005〜006 | 試験・実演 | サーバ再起動後に過去投稿がサーバから復元されないこと。接続中クライアントの一覧はセッション内の受信分に依拠する。 |
| SRS-NF-001 | 試験・計測 | 送信〜表示までの経過時間 |
| SRS-CSTR-001〜003 | 検査 | コード／構成に DB・ログイン機能がないこと |

### 9.2 トレーサビリティ（上流なしのため簡易）

上流要求文書がないため、現時点では **SRS ID ↔ 検証観点** のみを表で追跡する。StRS／SyRS が追加された場合は本文および表に上流 ID を追記する。
