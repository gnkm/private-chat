# 技術スタック

本書は [要件仕様書（SRS）](./requirements.md) に基づき、Private Chat の実装に用いる技術の選定と意図を記す。

## 1. 前提と方針

- **言語**: TypeScript をクライアント・サーバで共通化し、型とメッセージ形式の共有をしやすくする。
- **リアルタイム**: 要件の TBR-001 に対し、第一候補は **WebSocket（JSON ペイロード）**。双方向で投稿受付と配信を扱える一般的な構成とする。
- **永続化・認証**: DBMS・ユーザー登録・ログインは採用しない（SRS の制約に従う）。

## 2. クライアント（ブラウザ）

| 区分 | 技術 | 備考 |
|------|------|------|
| UI フレームワーク | React | Slack 風レイアウトの実装パターン・情報が豊富。 |
| 言語 | TypeScript | — |
| ビルド | Vite | 開発サーバと本番バンドルに広く用いられる。 |
| スタイリング | Tailwind CSS | レイアウトを素早く固定しやすい。 |
| 状態管理 | React（state / Context）。必要に応じて Zustand | 単一画面・単一会話のため、まずは最小構成を優先。 |
| リアルタイム | ブラウザ標準の WebSocket API | — |
| 表示名の保存 | `localStorage` 等 | SRS-UI-004（サーバアカウントと非紐付け）。 |

## 3. サーバ

| 区分 | 技術 | 備考 |
|------|------|------|
| ランタイム | Node.js（LTS） | — |
| HTTP | Express | 単一プロセス向けの定番。 |
| WebSocket | `ws`（Express では `express-ws` 等） | 単一ルームのブロードキャストに十分。 |
| メッセージ一覧の保持 | 各クライアントのメモリ（サーバは一覧を蓄積しない） | SRS-FUNC-005 / SRS-FUNC-006（再起動後はサーバから非復元）。 |
| 入力検証 | Zod | JSON ペイロードの検証と型の整合に利用可能。 |

## 4. パッケージ管理・品質ツール

| 区分 | 技術 | 備考 |
|------|------|------|
| パッケージマネージャ | **pnpm** | ワークスペース運用とディスク効率に向く。 |
| Lint / フォーマット | **Biome** | 単一ツールで lint と format を扱う。ESLint + Prettier の代わりに採用。 |
| 型検査 | TypeScript（`tsc`） | 厳格な設定を推奨。 |

## 5. テスト

| 区分 | 技術 | 備考 |
|------|------|------|
| 単体・コンポーネント | Vitest | Vite との親和性が高い。 |
| E2E | Playwright | 送信・配信・一覧更新の検証に利用可能。 |

## 6. 採用しないもの（要件との対応）

| 項目 | 理由 |
|------|------|
| RDB / NoSQL / Redis（永続ストアとして） | SRS-CSTR-001 |
| 認証・セッション基盤 | SRS-CSTR-002 |
| Socket.IO（初期段階） | 閉じた LAN・単一ルームでは素の WebSocket で十分。必要になれば再検討。 |

## 7. 未確定事項（SRS TBR との関係）

| ID | 内容 | 本書での位置づけ |
|----|------|------------------|
| TBR-001 | リアルタイム手段の最終選定 | 設計段階で WebSocket を既定とし、必要なら POST + SSE を代替として文書化する。 |

**解消済み**: TBR-002 は [SRS §5.4](./requirements.md) で定義（本文最大 12,000 文字、NUL 禁止）。サーバでは Zod 等で当該上限・禁止文字を検証する。
