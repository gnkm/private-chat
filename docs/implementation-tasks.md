## 2. 着手前に決めること（ブロッカー解消）

次を先に決めておくと実装とスキーマがブレない。

| 順 | 内容 | 根拠 |
|----|------|------|
| 1 | **TBR-002（解消済み）**: 本文は最大 **12,000** 文字（Unicode コードポイント）、**NUL（U+0000）のみ禁止**、LF／CR／タブは許可。Zod・UI の `maxLength` およびサーバ検証に反映する。 | [SRS §5.4](./requirements.md)（SRS-FUNC-007〜008） |
| 2 | **投稿 JSON のフィールド名（決定済み）**: 送信は `displayName`, `body`。配信の完成形は `id`, `displayName`, `body`, `sentAt`（ISO 8601 文字列）。`packages/shared` の Zod・型で一本化。 | [architecture](./architecture.md) §7.2 |
| 3 | **新規 WebSocket 接続時の履歴（決定済み）**: 接続直後にメモリ上の過去投稿を **再送しない**。クライアントは接続後のブロードキャスト分のみ表示する。 | [architecture](./architecture.md) §7.3 |
| 4 | **再接続ポリシー（決定済み）**: **指数バックオフ**あり。**ベース待機時間の上限は 30 秒**（ジッター前）。**ジッター**は **0〜500ms**（一様乱数）。**手動の再接続 UI は設けない**（自動再接続のみ。ページ再読み込みは任意）。**接続状態を示す専用 UI は設けない**。 | [architecture](./architecture.md) §7.3 |

**項 2 の補足（実装時の整理）**

- クライアント送信オブジェクトに `id` や `sentAt` を載せない。Zod は **送信用スキーマ** と **サーバ確定後スキーマ** を分け、サーバのみが `id` と `sentAt` を付与する。
- `id` は **string**（UUID 等の生成を推奨）。`sentAt` は **ISO 8601** の UTC 表記を推奨（例 `2026-04-18T12:34:56.789Z`）。将来メッセージ種別を増やす場合は、その時点でトップレベルに `type` 等のラッパーを足せばよい。

## 4. フェーズ 1 — `packages/shared`（契約層）

**目的**: Zod スキーマと TypeScript 型で **クライアント・サーバのメッセージ形式を共有**する（SRS-IF-001、TBR 解消後の検証）。

| 順 | タスク | 備考 |
|----|--------|------|
| 1.1 | クライアント送信ペイロード・サーバブロードキャスト用ペイロードの Zod スキーマと型を定義する。 | Vitest でスキーマの valid / invalid ケースを先に書く |
| 1.2 | 定数（本文最大 12,000 文字・NUL 禁止など [SRS §5.4](./requirements.md)）を shared に置く。 | SRS-FUNC-007〜008 |

**完了の目安**: サーバ・クライアント未実装でも、メッセージ形式の単体テストが緑になる。

## 5. フェーズ 2 — `apps/server`（Express + `ws` + プロセス内ストア）

**目的**: 投稿の受付、検証、単一ルームへのブロードキャスト、ID・タイムスタンプ付与、メモリ保持（SRS-FUNC-001〜006）。

| 順 | タスク | 備考 |
|----|--------|------|
| 2.1 | HTTP サーバの土台（Express）。ヘルスチェック用の軽い GET（運用・E2E の待機に使いやすい）。 | architecture §5 |
| 2.2 | WebSocket サーバ（`ws`）を同一プロセスにマウント。接続の増減を管理する。 | tech-stack §3 |
| 2.3 | 受信 JSON を Zod で検証。不正時は接続を乱暴に切らず、エラー応答の方針を決めてテストする。 | SRS-FUNC-001、SRS-IF-003 のサーバ側 |
| 2.4 | プロセス内ストア（配列または Map）に保存し、付与した `id` とサーバ時刻をブロードキャスト用メッセージに載せる。 | SRS-FUNC-003、005 |
| 2.5 | 接続中の全クライアントへブロードキャスト。新規接続には履歴を送らない（§2 項 3・architecture §7.3）。 | SRS-FUNC-002 |
| 2.6 | （任意）同時接続・簡易ストレステスト用のテストまたは手順メモ。 | SRS-NF-001 の目安 |

**完了の目安**: Vitest + 可能なら `ws` のモック／実ソケットで、送受信とブロードキャストが自動検証できる。

## 10. 推奨着手順序（一覧）

次の順で進めると **依存が少なく、早期に縦のスライス（共有型 → サーバ → クライアント → 統合 → E2E）** が切れる。

1. **§2** の前提を確認する（再接続ポリシー・新規接続時の履歴・投稿 JSON・本文上限はいずれも architecture / SRS にて確定済み）
2. **フェーズ 0** モノレポ・Biome・TS 基盤
3. **フェーズ 1** `packages/shared`
4. **フェーズ 2** `apps/server`（WebSocket + メモリ + 検証）
5. **フェーズ 3** `apps/web`（UI + WS クライアント）
6. **フェーズ 4** 開発プロキシ／本番静的配信
7. **フェーズ 5** Playwright E2E
8. **フェーズ 6** README・要件・アーキテクチャの追記・更新

---

## 11. 並行作業の目安

- **フェーズ 1** がスキーマを固めたあと、**サーバの受信処理** と **web の UI モック（WebSocket なし）** を薄く並行させることは可能。ただし **結合** はサーバのブロードキャストが動いてからが安全。
- **E2E** はフェーズ 4 まで待つと手戻りが少ない。
